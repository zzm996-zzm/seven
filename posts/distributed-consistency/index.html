分布式一致性不是玄学：从 CAP 到 Raft 的工程理解

用工程师的视角理解 CAP、BASE、两阶段提交和 Raft，不绕弯子

February 10, 2026 · 5 min read

分布式一致性这个话题被讨论得太多，但大部分文章要么太理论（CAP 定理证明），要么太碎片（随机讲一个场景），没有一个从工程角度把这些概念串起来的视角。这篇文章试图做这件事。

CAP 到底在说什么

CAP 是三个属性的首字母：

  • Consistency：每次读取都能看到最新写入（或错误）
  • Availability：每次请求都能得到非错误响应
  • Partition tolerance：节点间网络分区时系统仍然运行

CAP 的核心结论是：网络分区发生时，C 和 A 只能选一个。

注意这里有一个常被忽视的前提：只在网络分区发生时才需要选择。在分区没发生的正常情况下，C 和 A 是可以同时满足的。

BASE：对 ACID 的工程妥协

BASE 是实际分布式系统更常见的设计目标：

  • BAsically Available：基本可用，允许局部故障
  • Soft state：状态可以不同步，有短暂不一致
  • Eventually consistent：最终一致，而不是强一致

电商下单就是典型的 BASE 场景：库存扣减和订单创建可以先各自做，通过消息队列异步对账，最终达到一致。

两阶段提交（2PC）的问题

2PC 是最早的分布式事务方案：

Coordinator          Participant A    Participant B
    |                     |               |
    |---Prepare---------->|               |
    |---Prepare---------->|               |
    |<--Vote:Yes----------|               |
    |<--Vote:Yes--------------------------|
    |                     |               |
    |---Commit----------->|               |
    |---Commit---------------------------→|

问题很明显：Coordinator 挂了之后，Participant 处于锁定状态不知道该提交还是回滚，这就是阻塞问题。3PC 试图解决但引入了更复杂的状态机，实际工程中用得很少。

Raft：可理解的分布式共识

Raft 的设计目标就是"可理解性"。它把共识问题拆成三个相对独立的子问题：

  1. Leader 选举：谁来负责协调
  2. 日志复制：怎么保证所有节点日志一致
  3. 安全性：怎么保证提交的日志不会丢

关键规则只有几条：

- 有且只有一个 Leader（同一任期内）
- 所有写请求必须经过 Leader
- 日志必须在多数节点写入后才算提交
- 选举时只有拥有最新日志的节点才能当选

这几条规则组合起来，就能保证即使有少数节点宕机，系统仍然可用且一致。

工程落地的关键点

理解了理论，实际用 etcd / TiKV / CockroachDB 这些 Raft 系统时要注意：

  1. 读一致性：直接读 Follower 会读到旧数据，需要 linearizable read（通过 Leader 确认）
  2. 写入延迟：多数派确认带来额外 RTT，对延迟敏感的场景要评估
  3. 网络分区恢复：分区消除后落后节点的追赶（log catch-up）会带来短暂 CPU 尖刺

小结

  • CAP 告诉你分区时必须取舍，不是说永远只能选两个
  • BASE 是大多数互联网系统实际采用的一致性模型
  • 2PC 有阻塞问题，实际上更多用消息队列 + 事务消息代替
  • Raft 是目前最被广泛采用的共识算法，可读性高，实现相对完整
